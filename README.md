# Riverline Risk Intelligence Engine

*A borrower-lifecycle state machine for AI-driven debt collections.*

> "Study all these loan books to find a credit-segment we can uniquely disburse loans in" — *Riverline Manifesto*

Most "AI for collections" demos stop at "the bot can talk to a borrower." This project is a prototype of the data layer underneath: a system that takes raw collection call data and turns it into structured, auditable, decision-ready borrower intelligence. It doesn't just label individual calls; it models the borrower's **lifecycle** across multiple conversations over time, producing rollup metrics that a credit decision actually needs.

---

## 1. Why this project exists

Riverline's manifesto lays out a four-step master plan:
1. Replace call centers with AI agents to automate debt-collection.
2. Use these AI agents to own and resolve different loan books.
3. **Study all these loan books to find a credit-segment we can uniquely disburse loans in.**
4. Drive down loan operations with AI-flows and launch a bank with lower interest rates.

Step 3 is the core focus of this system. The data generated by those conversations is the product — it's how we eventually decide who to lend to.

This project is a prototype of that data layer: a system that takes raw collection call data (or WhatsApp messages) and turns it into structured, auditable, decision-ready borrower intelligence. It tracks a borrower across multiple calls over time, flags risk, measures promise reliability, checks agent compliance, and produces the kind of rollup metrics a credit decision needs.

## 2. Core Design Philosophy

### 2.1 Why a State Machine?
A simple classifier looks at one call and outputs a label. That's shallow. Collections is a *process* that unfolds over weeks and multiple contacts. A borrower who breaks one promise and then re-promises is in a hand-off position than one who's been ghosting for three calls straight, even if a single-call classifier labels both conversations as "promises". The state machine models the **lifecycle**, not the snapshot.

### 2.2 Why Constrained Transitions?
We do **not** ask the LLM "what state should this borrower be in?" with no guardrails. Instead, only a fixed set of next-states is valid from any current state. The LLM's job is narrowed to: *given this turn and the current state, which of the allowed transitions (if any) just occurred?*
* **Auditable:** You can trace why transitions occurred, avoiding arbitrary state jumps.
* **Compliance Posture:** Regulated fintech environments require predictable systems. Constrained transitions + a validation rail enforce this compliance.

### 2.3 Semantic Detection
Transitions rely on **semantic detection** (via LLMs) rather than rigid, fragile keyword matching. This allows the system to recognize hardship when a borrower says *"things have been tight since my hours got cut"* instead of strictly checking for *"I have a hardship"*.

### 2.4 The Fallback Chain & Local Caching: Groq → Gemini → Ollama
A single hardcoded LLM provider is a demo decision. A fallback chain and persistent caching are production decisions:
1. **Redis & Local File-based Caching:** Before calling external LLM APIs, the system queries Redis (if active) or falls back to a thread-safe local JSON cache (`data/llm_cache.json`). Repeat evaluations bypass the network entirely, resolving in <50ms.
2. **Throttled Network Fallback:** Throttling is applied only on cache misses, ensuring real network requests are spaced apart (3s delay) to avoid hitting rate limits.
3. **Groq (First):** Inference speed is the priority. In a real-time voice pipeline, latency is critical.
4. **Gemini (Second):** A highly capable fallback if Groq rate-limits or times out.
5. **Ollama (Third, Local):** A zero-dependency last resort that ensures local functionality without external API or internet access.

## 4. States, Transitions, and Risk Tiers

### 4.1 State Reference Table

| State | Meaning |
|---|---|
| `NEW` | Call/Contact started, no signals yet |
| `ENGAGED` | Borrower is responsive and conversing |
| `DISPUTING` | Borrower contests the debt amount or validity |
| `HARDSHIP_FLAGGED` | Borrower discloses genuine inability to pay (job loss, medical, etc.) |
| `PROMISE_MADE` | Borrower commits to a payment date or amount |
| `PROMISE_KEPT` | Confirmed payment matches the promise (resolved in next call context) |
| `PROMISE_BROKEN` | Confirmed no payment after promised date (resolved in next call context) |
| `ESCALATED` | Hostility, repeated evasion, or refusal to keep engaging |
| `GHOSTED` | Borrower disengages / call ends with no resolution |
| `RESOLVED` | Debt settled, restructured, or closed |

### 4.2 Transition Rules

* `NEW` ➜ `ENGAGED`, `ESCALATED`, `GHOSTED`
* `ENGAGED` ➜ `DISPUTING`, `HARDSHIP_FLAGGED`, `PROMISE_MADE`, `ESCALATED`, `GHOSTED`
* `DISPUTING` ➜ `ENGAGED`, `ESCALATED`, `RESOLVED`
* `HARDSHIP_FLAGGED` ➜ `PROMISE_MADE`, `RESOLVED`, `ESCALATED`
* `PROMISE_MADE` ➜ `PROMISE_KEPT`, `PROMISE_BROKEN`, `ESCALATED` *(Evaluated on next call)*
* `PROMISE_BROKEN` ➜ `PROMISE_MADE`, `ESCALATED`, `HARDSHIP_FLAGGED`
* `ESCALATED` ➜ `HARDSHIP_FLAGGED`, `RESOLVED`
* `GHOSTED` ➜ `ENGAGED`

### 4.3 Risk Tier Derivation (Green / Amber / Red)

The system automatically categorizes borrowers into three distinct risk tiers based on their final state machine position, regressions (broken promises/escalations), and contact histories:

* **🟢 Green Tier (Low Risk / Cooperative Collections)**:
  * *Derivation Rules:* The borrower's final state is in `{ENGAGED, PROMISE_MADE, PROMISE_KEPT, RESOLVED}` and `state_regressions == 0`.
  * *Interpretation:* The borrower is highly responsive, has never missed a promised payment date, and is cooperative. Standard automated reminder flows should be used.
* **🟡 Amber Tier (Medium Risk / Support & Restructuring)**:
  * *Derivation Rules:* The borrower's final state is `HARDSHIP_FLAGGED` (regardless of regressions) OR `state_regressions == 1`.
  * *Interpretation:* This tier routes borrowers to sympathetic, specialized agents rather than automated collectors. Genuine medical or financial hardship disclosures instantly anchor a borrower here, preventing them from falling into Red Tier. Alternatively, borrowers with exactly one broken promise are monitored here.
* **🔴 Red Tier (High Risk / Escalation & Manual Routing)**:
  * *Derivation Rules:* `state_regressions >= 2` OR the final state is `ESCALATED` OR the final state is `GHOSTED` with 2 or more distinct conversations.
  * *Interpretation:* Indicates high risk due to repeated broken payment commitments, active verbal hostility/refusal to pay, or multiple sequential contact attempts with zero engagement (ghosting). These accounts require manual agent intervention or legal escalation.

## 5. Development & Running the App

To run the application, you need to start two processes: the Python FastAPI backend and the Vite React frontend.

### Step 1: Install Dependencies (One-time Setup)
* **Backend:** Install the Python requirements from the root directory:
  ```bash
  pip install -r requirements.txt
  ```
* **Frontend:** Install Node packages from the `dashboard` directory:
  ```bash
  cd dashboard
  npm install
  cd ..
  ```

### Step 2: Configure Environment Keys
Copy `.env.example` to `.env` in the root directory and add your API keys:
```env
GROQ_API_KEY=your-groq-key
GEMINI_API_KEY=your-gemini-key
```

### Step 3: Run the Backend API Server
Start the FastAPI server from the root directory:
```bash
python src/server.py
```
This runs the server at **`http://localhost:8000`** with auto-reload configured to watch only the `src` directory (so dynamic file writes don't crash or restart your server).

### Step 4: Run the Web Dashboard
In a **new terminal window**, navigate to the `dashboard` folder and start the Vite development server:
```bash
cd dashboard
npm run dev
```
Open **`http://localhost:5173`** (or the port specified in Vite terminal) in your browser to view the interactive portfolio dashboard.

### Step 5: (Optional) Pre-Compile Caches & Baselines
To process the baseline 8 borrowers offline and populate your local API query cache (`data/llm_cache.json`) for instant dashboard load speeds, run:
```bash
python src/run_engine.py
```

## 6. Dashboard Playground & Raw Script Parsing

The dashboard's **Simulation Console (Playground)** supports evaluating both single turns and full conversation logs:
* **Raw Script Mode:** Paste a multi-turn conversation script. E.g.:
  ```text
  Agent: Hi Priya, this is Riverline.
  Borrower: I lost my job last week, things are tight.
  Agent: I understand. We can set a restructured plan.
  Borrower: I will pay ₹1,000 next Friday.
  ```
* **Real-time LLM Analysis:** If the Python API server (`http://localhost:8000`) is running, the dashboard submits the script to the backend, which parses the text, triggers state engine classification, runs it through the validation rail, and returns full metrics and state transition timelines.
* **Fallback Simulation Mode:** If the backend is offline, the React app uses a client-side parser and rule-based classifier to simulate transitions.
* **Structured Export:** You can export the resulting structured JSON directly from the UI.

### 6.1 Multi-Day / Multi-Call Simulator Syntax

To simulate multiple calls over several days or different communication channels, separate the calls using one of the following marker styles:

* **Style A (Call Headers):** `=== Call <number>: <channel> (<date>) ===`
  ```text
  === Call 1: whatsapp (2026-04-02) ===
  Agent: Hello Vikram...
  Borrower: I will pay by Friday.

  === Call 2: voice (2026-04-07) ===
  Agent: We did not receive Friday's payment...
  ```
* **Style B (Date Headers):** `--- Date: YYYY-MM-DD, Channel: channel ---`
  ```text
  --- Date: 2026-04-02, Channel: whatsapp ---
  Agent: Hi Priya...
  ```
* **Style C (Compact Headers):** `[YYYY-MM-DD, channel]`
  ```text
  [2026-04-02, voice]
  Agent: Hello Sunita...
  ```

If no separator is defined, the script defaults to a single voice conversation on date `2026-06-21`. The state engine maintains state variables (like current state and regressions count) across call boundaries.
